# ADR-005 : Gestion des données

**Statut** : Accepté  
**Date** : 2026-07-22  
**Approbation** : Équipe

## Contexte
Les pages publiques doivent afficher des données réelles depuis Supabase. Le projet nécessite une organisation claire des appels à la base de données.

## Décisions

### Organisation des services
- Chaque table Supabase a son propre module dans `src/lib/<table>.ts`
- Fonctions nommées et exportées individuellement (pas de classes)
- Retour : `Promise<T>` pour les requêtes simples, `Promise<{ data: T[]; count: number }>` pour les requêtes paginées
- Les erreurs Supabase sont propagées par `throw` — la gestion est déléguée aux pages via `try/catch`

### Tables et responsabilités
| Table | Fichier service | Opérations |
|-------|----------------|------------|
| `players` | `src/lib/players.ts` | getAll, getById, create, update, remove |
| `matches` | `src/lib/matches.ts` | getAll, getPast (paginé), getUpcoming, getById, create, update, remove |
| `news` | `src/lib/news.ts` | getAll (paginé), getRecent, getBySlug, create, update, remove |
| `staff` | `src/lib/staff.ts` | getAll, getById, create, update, remove |

### Pagination
- Pagination serveur : `.range(from, to)` + `count: 'exact'` de Supabase
- Matchs : 12 par page ; News : 5 par page
- `getUpcomingMatches()` sans pagination (liste courte)

### Types
- Types centralisés dans `src/types/index.ts`
- Utilisation de `type` (pas `interface`) pour correspondre aux retours Supabase
- Champs optionnels marqués `| null` (pas `?`)

## Conséquences
- Séparation claire des responsabilités
- Réutilisabilité des services entre pages
- Facilité de test unitaire
- Pas de duplication des appels Supabase
