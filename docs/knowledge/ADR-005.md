# ADR-005 : Gestion des données

**Statut** : Accepté  
**Date** : 2026-07-22  
**Approbation** : Équipe

## Contexte
Les pages publiques doivent afficher des données réelles depuis Supabase. Le projet utilise déjà un client Supabase configuré dans `src/lib/supabase.ts`.

## Décisions

### Organisation des services
- Un module par entité dans `src/lib/` : `players.ts`, `matches.ts`, `news.ts`, `staff.ts`
- Chaque module expose des fonctions pures nommées (pas de classes)
- Les fonctions sont async et utilisent le client `supabase` centralisé

### Pattern de retour
- Les fonctions retournent directement les données typées (`Promise<T[]>` ou `Promise<{ data: T[]; count: number }>`)
- Les erreurs sont propagées par `throw` — la gestion est déléguée aux composants/pages
- Exception : `getNewsBySlug` et `getPlayerBySlug` retournent `T | null` pour les recherches unitaires

### Pagination
- Pagination serveur via `.range(from, to)` + `.select('*', { count: 'exact' })`
- Matchs : 12 par page ; News : 5 par page (selon spécification)
- `getUpcomingMatches()` : pas de pagination (liste courte), limite configurable
- `getPastMatches()` et `getAllMatches()` : paginées

### Sécurité
- RLS activée sur toutes les tables (voir `supabase/migrations/`)
- Lecture publique pour les données actives/publiées
- Filtres systématiques : `.eq('is_active', true)`, `.eq('status', 'published')`

## Conséquences
- Séparation claire des responsabilités (lecture seule pour le public)
- Réutilisabilité des services dans les pages et composants
- Facilité de test : chaque fonction peut être testée isolément
- La pagination serveur évite les problèmes de performance avec de gros volumes
