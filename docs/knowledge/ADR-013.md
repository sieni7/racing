# ADR-013 : Administration CRUD — Sprint 3

**Statut** : Accepté  
**Date** : 22 juillet 2026  
**Approbation** : Équipe

## Contexte
Le Sprint 3 construit l'interface d'administration complète pour gérer les quatre entités principales : Joueurs, Matchs, Actualités et Staff.

## Décisions

### Architecture MVC — Contexte Admin
- **Modèle** : Services Supabase dans `src/lib/*.ts` (players, matches, news, staff) — retour direct des données, `throw` sur erreur.
- **Vue** : Composants réutilisables `AdminTable`, `AdminForm`, `ConfirmModal` + pages `Dashboard`, `Players`, `Matches`, `News`, `Staff`.
- **Controller** : `AdminContext` (`src/contexts/AdminContext.tsx`) — unique point d'entrée pour les actions CRUD, gère le state global (players, matches, news, staff) et expose `fetch*`, `create*`, `update*`, `delete*`.

### Layout Admin
- `AdminLayout` avec sidebar pliable (icônes + labels), header, navigation protégée par `PrivateRoute` (admin only).
- Routes imbriquées sous `/admin` : `/` (Dashboard), `/players`, `/matches`, `/news`, `/staff`.

### Composants Réutilisables
| Composant | Responsabilité |
|-----------|----------------|
| `AdminTable` | Tableau paginé, triable, recherche multi-champs, actions edit/delete |
| `AdminForm` | Modal formulaire générique (text, number, date, select, textarea, file) + validation required + erreurs |
| `ConfirmModal` | Modale confirmation suppression avec état loading |
| `Skeleton` | Variantes `CardSkeleton`, `ListSkeleton` pour loading |

### Services Supabase — Pattern Unifié
- Toutes les fonctions `throw` sur erreur Supabase, retournent la donnée directe (`T` ou `T[]`).
- Suppression du wrapper `toResult()` (héritage Sprint 2).
- `players.ts` aligné sur le pattern `throw` (cohérence avec matches, news, staff).

### Auth & Sécurité
- Routes admin protégées par `PrivateRoute` (redirect `/login` si non connecté, `/` si non admin).
- Rôle admin lu depuis `user.user_metadata.role` (sync via trigger Supabase).

## Conséquences
- ✅ Architecture MVC respectée dans l'admin (séparation Model/View/Controller).
- ✅ Composants DRY — `AdminTable`/`AdminForm` réutilisés sur 4 entités.
- ✅ API services cohérente — pattern `throw` uniforme.
- ✅ Code splitting — chaque page admin = chunk séparé.
- ⚠️ Tests unitaires nécessitent mise à jour des mocks (nouveau pattern `throw`).
