# ADR-012 : Pages publiques et données dynamiques — Sprint 2

**Statut** : Accepté  
**Date** : 2026-07-22  
**Approbation** : Équipe

## Contexte
Le Sprint 2 ajoute 4 services Supabase, 4 composants UI réutilisables, et 4 pages publiques dynamiques pour le site vitrine du Racing Club de Bingerville.

## Décisions

### Pattern Service
- Chaque table Supabase a son propre module `src/lib/<table>.ts` avec des fonctions exportées pures.
- Les fonctions retournent `Promise<T>` ou `Promise<{ data: T[]; count: number }>` pour la pagination.
- Les erreurs Supabase sont propagées (throw) — la gestion est déléguée aux pages via try/catch.

### Types centralisés
- `src/types/index.ts` regroupe tous les types (Player, Match, NewsItem, Staff).
- Les types sont des `type` (pas `interface`) pour rester cohérent avec les réponses Supabase.
- Les champs optionnels sont marqués `| null` (pas `?`) pour correspondre aux retours Supabase.

### Pagination
- Approche : pagination serveur via `.range(from, to)` + `count: 'exact'`.
- Matchs : 12 par page ; News : 9 par page.
- `getUpcomingMatches()` sans pagination (liste courte, limit 5 pour la home, 50 pour la page matchs).
- `getPastMatches()` et `getAllMatches()` paginés.

### Composants UI
- Atomiques : un composant = une carte (PlayerCard, MatchCard, etc.).
- Squelettes de chargement dédiés (Skeleton, CardSkeleton, ListSkeleton) plutôt que du inline.
- Images : `loading="lazy"` par défaut ; `onError` pour fallback sur les initiales/placeholder.

### Gestion des erreurs
- Distinction 404 (PGRST116) vs erreur réseau dans `NewsArticlePage` : le code d'erreur Supabase est inspecté pour afficher un message approprié.

## Conséquences
- Facilité de test : chaque service peut être testé indépendamment.
- La pagination serveur évite le chargement excessif en mémoire pour les pages matchs et news.
- L'absence d'appelant pour `getPlayerBySlug` a mené à sa suppression (ADR-202).
- Les migrations SQL sont dans `supabase/migrations/` pour la reproductibilité.
